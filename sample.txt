## 1. Overview

This document outlines the plan to design and deliver a structured API automation framework for our services. The framework will cover four layers of test coverage — Smoke, Contract, CRUD Lifecycle, and Security/Negative tests — integrated into the CI/CD pipeline with reporting and alerting.

Given that I am contributing to this initiative while actively supporting two teams (2–3 days available per week), this roadmap is structured to deliver value incrementally, starting with a working foundation and expanding progressively.

---

## 2. Goals

- Catch regressions and deployment failures early through automated API tests
- Validate API responses against the OpenAPI 3.0 spec (openapi-dev.yaml) to prevent schema drift
- Ensure full CRUD data lifecycle correctness for stateful resources
- Reject invalid and unauthorized requests with correct status codes and safe error messages
- Provide self-contained HTML reports as GitHub Actions artifacts per run
- Gate the CD pipeline from promoting broken builds to the next environment

---

## 3. Technology Stack

| Component | Tool / Library |
|---|---|
| Language | Java |
| API Testing | RestAssured |
| Test Runner | TestNG |
| Contract Validation | Atlassian swagger-request-validator |
| Assertions | AssertJ |
| Reporting | Allure + allure-testng |
| Build Tool | Maven (existing stack) |
| Slack Alerts | Custom TestNG Listener via HTTP POST |
| CI/CD | GitHub Actions |

**Rationale:** Java was chosen to align with the team's existing skills and Spring/Maven stack, minimizing onboarding effort and enabling shared code review standards.

---

## 4. Framework Architecture (High-Level)

```
api-automation/
├── pom.xml
├── src/test/java/
│   ├── base/          → BaseTest.java (auth, RestAssured config)
│   ├── smoke/         → SmokeTests.java
│   ├── contract/      → ContractTests.java
│   ├── crud/          → Per-resource lifecycle tests
│   ├── security/      → Negative & auth tests
│   └── listeners/     → SlackNotifier.java (TestNG listener)
└── src/test/resources/
    ├── openapi-dev.yaml
    └── testng.xml
```

The framework is layered so each suite can be run independently or as a full suite. Tests are fully isolated — each test creates and cleans up its own data.

## 5. Phased Delivery Roadmap

> Working days available: **2–3 days per week**

---

### Phase 1 — Foundation & Smoke Tests
**Duration:** 2 weeks (4–6 working days)  
**Deliverable:** A working Maven project with smoke tests running in GitHub Actions

**What will be done:**
- Set up Maven project structure with RestAssured, TestNG, and AssertJ dependencies
- Create `BaseTest.java` — shared auth token setup, base URL config, environment switching
- Write smoke tests for all major endpoints (GET requests, status code + top-level shape assertions)
- Wire up GitHub Actions workflow (`.github/workflows/api-tests.yml`)
- Upload basic Surefire/Allure report as GitHub Actions artifact

**What leadership can expect to see:**
- A runnable test project in the repository
- Green/red test results visible in GitHub Actions on every run
- A downloadable HTML report per run

---

### Phase 2 — Contract Tests
**Duration:** 2 weeks (4–6 working days)  
**Deliverable:** Schema validation running against all endpoints using openapi-dev.yaml

**What will be done:**
- Integrate `swagger-request-validator` library
- Load `openapi-dev.yaml` once per test session
- Write contract tests for each endpoint — validate real responses field-by-field against the spec
- Capture and report schema violations (missing fields, type mismatches, undeclared fields, enum violations)

**What leadership can expect to see:**
- Any schema drift between the API and the spec is automatically caught on every run
- Contract violations shown clearly in the Allure report

---

### Phase 3 — CRUD Lifecycle Tests
**Duration:** 3 weeks (6–9 working days)  
**Deliverable:** End-to-end data lifecycle tests for all stateful resources (DynamoDB)

**What will be done:**
- Identify all stateful resources (e.g. Orders, Users, etc.)
- Write full Create → Read → Update → Delete → Verify lifecycle tests per resource
- Implement `@AfterMethod(alwaysRun = true)` cleanup to guarantee data is removed even on test failure
- Validate data persistence, correct status codes, and response correctness at each step

**What leadership can expect to see:**
- Confidence that real data flows correctly through all API operations end-to-end
- No orphaned test data left in DynamoDB after runs

---

### Phase 4 — Security & Negative Tests
**Duration:** 2 weeks (4–6 working days)  
**Deliverable:** Negative test coverage for auth, permissions, invalid inputs, and safe error messages

**What will be done:**
- Write parameterized tests covering: missing/expired tokens (401), wrong role (403), invalid inputs (400), non-existent resources (404), oversized payloads (413)
- Assert that error responses never contain internal details (stack traces, DB names, table names)
- Cover injection-style inputs in query params and request bodies

**What leadership can expect to see:**
- Verified proof that the API rejects bad actors correctly
- Safe error message compliance confirmed on every run

---

### Phase 5 — Reporting, Alerting & Pipeline Gating
**Duration:** 1 week (2–3 working days)  
**Deliverable:** Full Allure HTML reports, Slack failure alerts, and CD pipeline exit code gating

**What will be done:**
- Finalize Allure report configuration — self-contained HTML uploaded as GitHub Actions artifact per run
- Implement `SlackNotifier` TestNG listener — posts to Slack channel on failure with environment, failed count, and artifact link
- Confirm Maven Surefire exit code gating blocks CD pipeline promotion on test failure

**What leadership can expect to see:**
- Instant Slack alerts when tests fail in any environment
- CD pipeline automatically blocked from promoting broken builds
- Full downloadable HTML report for every run
