## 2. Goals

- Catch regressions and deployment failures early through automated API tests
- Validate API responses against the OpenAPI 3.0 spec to prevent schema drift
- Ensure full CRUD data lifecycle correctness for stateful resources
- Reject invalid and unauthorized requests with correct status codes and safe error messages
- Provide self-contained HTML reports as GitHub Actions artifacts per run
- Gate the CD pipeline from promoting broken builds to the next environment

## 5. Phased Delivery Roadmap

---

### Phase 1 — Smoke Tests *(First Real Deliverable)*
**Duration: 2.5–3 weeks (~17–25 hours)**

This is the starting phase. Maven and TestNG are already known, so the focus is on learning RestAssured, AssertJ, and GitHub Actions while simultaneously building the first real tests.

**Learning required in this phase:**

| Topic | Estimated Hours |
|---|---|
| RestAssured basics — `given().when().then()` syntax, reading responses | 4–5 hours |
| AssertJ — `assertThat()` for status codes and response body assertions | 1–2 hours |
| GitHub Actions — workflow file, running `mvn test`, uploading artifact | 3–4 hours |

**What will be built:**

- `BaseTest.java` — RestAssured base URI config, shared auth token setup, environment switching via `.properties` file
- Smoke tests for all major endpoints — assert status code + top-level response shape
- Immediate cleanup for any write operations used in smoke tests
- GitHub Actions workflow (`.github/workflows/api-tests.yml`) — triggers on every push
- Basic Allure or Surefire HTML report uploaded as GitHub Actions artifact per run

**Smoke test example structure:**
```
GET /users      → assert 200, body has "users" key, value is a list
GET /orders     → assert 200, body has "orders" key, value is a list
POST /orders    → assert 201, body has "id" key → immediately DELETE created record
```

**Deliverable at end of phase:**
- Smoke tests running automatically on every push via GitHub Actions
- Pass/fail visible in GitHub Actions
- Downloadable HTML report per run

**What leadership sees:** Real automated tests running in CI. First tangible proof of value.

---

### Phase 2 — Contract Tests
**Duration: 4–5 weeks (~30–40 hours)**

Contract testing introduces a new library (`swagger-request-validator`) and a new concept — validating real API responses field-by-field against the OpenAPI 3.0 spec. Extra time is allocated for the learning curve.

**Learning required in this phase:**

| Topic | Estimated Hours |
|---|---|
| OpenAPI 3.0 schema concepts — required fields, types, enums | 2–3 hours |
| swagger-request-validator — loading yaml, validating responses | 4–5 hours |
| Understanding and interpreting schema violation reports | 2–3 hours |

**What will be built:**

- Load `openapi-dev.yaml` once per test session (not per test — performance matters)
- Contract tests per endpoint — real responses unmarshalled and validated against the declared schema
- Clear reporting of violations: type mismatches, missing required fields, undeclared fields, enum violations

**Deliverable at end of phase:**
- Schema drift between the API and spec is automatically caught on every CI run

**What leadership sees:** Schema compliance enforced automatically. No more silent contract breaks.

---

### Phase 3 — CRUD Lifecycle Tests
**Duration: 5–6 weeks (~40–50 hours)**

This is the most complex phase. Tests write real data to DynamoDB, and test isolation and guaranteed cleanup must be handled carefully.

**Learning required in this phase:**

| Topic | Estimated Hours |
|---|---|
| `@AfterMethod(alwaysRun = true)` — guaranteed cleanup even on failure | 1–2 hours |
| Chaining RestAssured requests — using IDs from one call in the next | 2–3 hours |
| Safe test data handling in a shared environment | 2–3 hours |

**What will be built:**

- Full Create → Read → Update → Delete → Verify lifecycle per stateful resource
- Guaranteed data cleanup after every test, even on failure
- Validation of data persistence, correct status codes, and response body correctness at each step

**Lifecycle flow per resource:**
```
POST  /orders           → Create   → assert 201, save id
GET   /orders/{id}      → Read     → assert 200, data matches
PUT   /orders/{id}      → Update   → assert 200, field changed
GET   /orders/{id}      → Verify   → assert updated value present
DELETE /orders/{id}     → Delete   → assert 204
GET   /orders/{id}      → Confirm  → assert 404
[teardown]              → Cleanup  → runs always, even if test failed mid-way
```

**Deliverable at end of phase:**
- End-to-end data lifecycle verified automatically in CI
- No orphaned test data left in DynamoDB after any run

**What leadership sees:** Full confidence in data correctness across all API operations.

---

### Phase 4 — Security & Negative Tests
**Duration: 3–4 weeks (~25–30 hours)**

**Learning required in this phase:**

| Topic | Estimated Hours |
|---|---|
| TestNG `@DataProvider` — parameterized tests for many input variations | 2–3 hours |
| Asserting on error response bodies safely | 1–2 hours |
| Understanding "no internal data leaked" and how to verify it | 1–2 hours |

**What will be built:**

- Parameterized negative tests covering:
  - Missing / expired token → 401
  - Wrong role / insufficient permission → 403
  - Invalid input values → 400
  - Non-existent resource → 404
  - Oversized payload → 413
- Assertions that error bodies never contain stack traces, DB names, table names, or internal details

**Deliverable at end of phase:**
- API rejection behavior fully verified in CI
- Safe error message compliance confirmed on every run

**What leadership sees:** Verified proof that the API correctly rejects bad actors and never leaks internals.

---

### Phase 5 — Reporting, Slack Alerts & Pipeline Gating
**Duration: 2–3 weeks (~15–20 hours)**

**What will be built:**

- Allure HTML report finalized — self-contained, uploaded as GitHub Actions artifact per run
- `SlackNotifier` TestNG listener — posts to Slack channel on failure with environment name, failed count, and artifact link
- Maven Surefire exit code gating confirmed — non-zero exit on failure blocks CD pipeline from promoting to next environment

**Deliverable at end of phase:**
- Instant Slack alerts on any test failure
- CD pipeline automatically blocked on broken builds
- Full downloadable HTML report for every run

**What leadership sees:** Complete reporting and pipeline safety net fully operational.
